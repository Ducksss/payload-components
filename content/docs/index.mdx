Payload Kits distributes installable Payload CMS blocks through a public registry and a small wrapper CLI.

The v2 app is intentionally not a Payload CMS site. It is a Fumadocs app with MDX documentation, static registry assets, and package tooling for validating installs into external Payload projects.

## What changed

- The website no longer boots Payload admin routes, collections, globals, database adapters, or generated Payload app types.
- Documentation lives in `content/docs` and is compiled by Fumadocs MDX.
- Registry source lives outside the app runtime and builds to `public/r`.
- Payload remains the target ecosystem, not the site runtime.

## Current install

```bash
npx payload-kit add hero-basic
```

The CLI verifies project shape, installs the registry item, applies Payload layout fragments, runs post-install tasks, and records install state.

## Current kits

- `hero-basic`: headline, description, CTA links, and proof badges.
- `feature-grid-basic`: repeatable feature cards and optional CTA wiring.

## Public registry

The canonical public registry template is:

```bash
https://payload-components.xyz/r/{name}.json
```

The current public files are:

- `https://payload-components.xyz/r/registry.json`
- `https://payload-components.xyz/r/hero-basic.json`
- `https://payload-components.xyz/r/feature-grid-basic.json`

`payload-kits.com` is not the active registry domain. Keep public registry access unauthenticated.
