Payload Kits currently targets Payload v3 projects using Next.js App Router and the supported website-style block layout shape.

## Requirements

<Callout type="warn">
  The installer verifies the target project shape **before** it mutates anything.
  If a requirement is missing, it stops rather than leaving a half-wired repo.
</Callout>

Your project must run these majors:

| Dependency | Version       |
| ---------- | ------------- |
| `payload`  | v3            |
| `next`     | v15 or v16    |

…and it must already contain the website-style block layout the installer patches into:

<Files>
  <File name="components.json" />
  <Folder name="src" defaultOpen>
    <File name="payload.config.ts" />
    <Folder name="blocks" defaultOpen>
      <File name="RenderBlocks.tsx" />
    </Folder>
    <Folder name="collections/Pages" defaultOpen>
      <File name="index.ts" />
    </Folder>
  </Folder>
</Files>

Plus `package.json` scripts for `generate:types` and `generate:importmap`, which the CLI runs after wiring.

## Install a kit

Pick a kit and run its `add` command from the root of your Payload project:

<Tabs items={['hero-basic', 'feature-grid-basic']}>
  <Tab value="hero-basic">
    ```bash
    npx payload-kit add hero-basic
    ```
  </Tab>
  <Tab value="feature-grid-basic">
    ```bash
    npx payload-kit add feature-grid-basic
    ```
  </Tab>
</Tabs>

## What the CLI does

The wrapper CLI owns the Payload-specific work that direct registry installation cannot handle:

<Steps>
  <Step>
    **Validate the target project shape** — refuse to touch a repo that isn't a supported Payload v3 + Next.js project.
  </Step>
  <Step>
    **Build or resolve the registry item** and **install source files** through the shadcn registry flow.
  </Step>
  <Step>
    **Apply Payload layout and render fragments** — register the block in `RenderBlocks` and the `Pages` layout field.
  </Step>
  <Step>
    **Install missing kit dependencies** so the block compiles in your project.
  </Step>
  <Step>
    **Run post-install scripts** (`generate:types`, `generate:importmap`) and **write `.payload-kit/state.json`**.
  </Step>
</Steps>

## Idempotency

<Callout type="success">
  Running the same install twice should not duplicate imports, block registrations,
  or layout entries. The installer verifies files, fragments, dependencies, and
  recorded state before deciding whether any work is needed.
</Callout>

## Direct registry installs

Direct shadcn installs can deliver files, but they do not apply Payload-specific fragments or run the complete post-install workflow. Use `payload-kit add` for real target projects.
