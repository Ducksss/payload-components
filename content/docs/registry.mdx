---
title: Registry Contract
description: How the public registry, manifests, and wrapper CLI work together.
---

Payload Kits uses two layers:

- a shadcn-compatible public registry for file delivery
- a wrapper manifest for Payload-specific install behavior

## Public registry

The public registry is generated from `payload-kits/registry.json`:

```bash
pnpm registry:build
```

The generated files are emitted to `public/r` and can be served statically.

The active public registry is:

```bash
https://payload-components.xyz/r/{name}.json
```

Current URLs:

```bash
curl -fsSL https://payload-components.xyz/r/registry.json
curl -fsSL https://payload-components.xyz/r/hero-basic.json
curl -fsSL https://payload-components.xyz/r/feature-grid-basic.json
```

The root registry must list item metadata and file targets without embedding `files[].content`. Each item JSON must include installable file content. `pnpm test:registry` enforces that local generated artifacts are reproducible from source.

## Manifests

Each wrapper manifest in `payload-kits/manifests` defines:

- supported Payload and Next.js majors
- files expected after install
- peer dependency requirements
- Payload fragments to apply
- post-install scripts
- recovery metadata for patched files
- sample content for docs and testing

## Manifest v1

The v1 public authoring contract is intentionally small:

- kit identity: `name`, `version`, `title`, `description`, and `registryItemName`
- compatibility: `supportedTargets`, `supports.payloadMajors`, and `supports.nextMajors`
- package requirements: `dependencies` and `peerDependencies`
- file ownership: `files` paths that the generated registry installs into the target project
- Payload wiring: `payloadFragments` for `RenderBlocks` and `Pages` layout registration
- recovery: `recovery.patchedFiles` for host files that require manual review during repair
- post install: `postInstall` scripts, currently `generate:types` and `generate:importmap`
- docs/testing metadata: `preview.summary` and `sampleContent`

Invalid manifests fail before install with an error that names the invalid field.

## Support matrix

`payload-kits/support-matrix.json` defines the target project shapes the installer understands.

The current alpha support matrix is intentionally narrow. New project shapes should be added only when the installer can verify them with tests.

## Recovery state

The CLI records install state in `.payload-kit/state.json` inside the target project. That state is not a substitute for disk verification; the installer still checks actual files and fragments before declaring a kit installed.

`payload-kit doctor --json` is the CI-safe recovery view. It returns exit `1` for recoverable drift and includes a retry command for the affected kit.
