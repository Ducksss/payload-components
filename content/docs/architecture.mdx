Payload Kits is two projects living in one repository — and they deliberately never share a Payload runtime.

<Cards>
  <Card icon={<Boxes />} title="Site runtime">
    A Fumadocs + Next.js app: the landing, these docs, and the kit catalog. Boots
    with no database, no `PAYLOAD_SECRET`, and no admin.
  </Card>
  <Card icon={<Workflow />} title="Kit runtime">
    The Payload block code the CLI installs into _your_ repo. It only ever runs
    there, against real Payload project shapes.
  </Card>
</Cards>

The two responsibilities are:

1. Run a documentation and catalog site.
2. Build, validate, and distribute Payload kit installs.

Those responsibilities should not share a Payload runtime.

## Site runtime

The site runtime is Fumadocs on Next.js. Everything it needs lives in the app tree:

<Files>
  <Folder name="src/app" defaultOpen>
    <File name="api/search/route.ts" />
  </Folder>
  <Folder name="content/docs" defaultOpen>
    <File name="*.mdx" />
  </Folder>
  <File name="source.config.ts" />
  <File name="src/lib/source.ts" />
</Files>

- `src/app` contains the app router routes.
- `content/docs` contains MDX documentation.
- `source.config.ts` defines the Fumadocs MDX collection.
- `src/lib/source.ts` exposes the loaded docs source.
- `src/app/api/search/route.ts` exposes Fumadocs search.

<Callout type="success">
  No Payload admin, collection config, global config, database adapter, or Payload
  local API is required for the site to boot.
</Callout>

## AI-readable surfaces

The site exposes the same docs in browser and machine-readable formats:

- `/llms.txt` publishes the concise source map for AI crawlers.
- `/llms-full.txt` publishes the compiled documentation body.
- `/llms.mdx/docs/**/content.md` publishes each docs page as markdown.
- JSON-LD marks the site, CLI, FAQ, kit catalog, and docs articles for structured extraction.

## Kit runtime

The kit runtime is external. A kit installs into a supported Payload v3 + Next.js project, where the Payload-specific files actually live:

- block configs
- frontend block components
- `RenderBlocks` registration fragments
- `Pages` collection layout fragments
- `generate:types` and `generate:importmap` post-install tasks

## Registry source

The shadcn-compatible registry reads source files from `payload-kits/source` and emits public JSON to `public/r`.

That keeps Payload-specific kit source out of the Fumadocs app tree while preserving install targets like `~/src/blocks/HeroBasic/Component.tsx`.

## Kit families and shared code

A component family ships one installable kit per structural variant — `hero-basic`, `hero-video`, `hero-dramatic` — each its own registry item and manifest. Variant selection is the catalog and the install command, never an interactive prompt. Content-level variation (optional or extra editor fields) lives in the Payload block config, not a separate kit.

Variants that share fields or markup reference a common source file under `payload-kits/source/blocks/shared`, installed once into the consumer repo — for example `~/src/blocks/shared/heroFields.ts` — and composed by each variant config as `fields: [...heroFields, /* variant-specific */]`. Editing the shared file updates every installed variant; re-running an install never overwrites an edited copy. Internal shared modules ship as registry files, not `registryDependencies` (which resolve only public shadcn components).

## Why this split matters

<Callout type="idea">
  A docs site should build without a database, Payload secret, or admin import map.
  A Payload kit should still be tested against real Payload project shapes. Splitting
  the app runtime from the install target keeps both jobs honest.
</Callout>
