Payload Kits is open-source and community-first — adoption and contributions are the whole point.

<Callout type="idea">
  Aim every contribution at making the **install contract clearer**, not just
  adding more UI. A kit that installs cleanly and twice is worth more than a
  prettier block that drifts.
</Callout>

## New kit checklist

A kit proposal should arrive complete — incomplete kits don't ship:

<Steps>
  <Step>
    **A small, useful Payload block concept** — narrow enough to verify, useful enough to install.
  </Step>
  <Step>
    **A block config** with editor-friendly fields.
  </Step>
  <Step>
    **A frontend component** that renders without extra app assumptions.
  </Step>
  <Step>
    **A manifest** with files, fragments, dependencies, and sample content.
  </Step>
  <Step>
    **Install tests** for both first install and repeated install.
  </Step>
  <Step>
    **Documentation** explaining the target use case.
  </Step>
</Steps>

## Keep the catalog curated

The alpha catalog should stay small enough to verify.

<Callout type="success">
  A kit is ready when it installs cleanly, can be run twice, and has docs good
  enough for another developer to reason about the target project changes.
</Callout>

## Report install drift

If a supported Payload project shape changes, open an issue with enough context for the wrapper to update the support matrix — or reject the target shape clearly:

- Payload version
- Next.js version
- the failing command
- relevant `RenderBlocks` and `Pages` collection snippets
- the expected install result
